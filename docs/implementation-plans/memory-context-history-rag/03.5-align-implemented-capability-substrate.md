# 3.5. Align Implemented Capability Substrate

## Goal

Align the status, configuration, diagnostics, and test substrate implemented in
phases 1 through 3 before adding conversation history, persistence, memory, RAG,
or research behavior.

This phase is a correction gate. It should remove ownership drift from the
already-landed work and from the remaining plan files so later phases do not
build on the wrong package boundary.

## Why Before Four

Phases 1 through 3 created the visible control plane: capability status, service
configuration, and app-path tests. That substrate will steer every later
capability. If it mixes portable core contracts with service adapter choices,
the later history, memory, RAG, and research work will repeat that mistake in
larger code paths.

Run this phase after phase 3 and before `04-conversation-history-context.md`.

## Ownership Rules To Re-Assert

| Concern                                  | Owner                                                                                    |
| ---------------------------------------- | ---------------------------------------------------------------------------------------- |
| Portable capability config contracts     | `packages/partner-ai-core/src/domain/capabilities/contracts/capability-configuration.ts` |
| Host capability manifest contracts       | `packages/partner-ai-core/src/domain/capabilities/contracts/capabilities.ts`             |
| Context, memory, RAG, and research ports | `packages/partner-ai-core/src/ports/context/**`                                          |
| Turn policy and prepared context         | `packages/partner-ai-core/src/application/stream-chat/**`                                |
| Env parsing and deployable mode parsing  | `apps/partner-ai-service/src/config/**`                                                  |
| Concrete mode selection and diagnostics  | `apps/partner-ai-service/src/composition/capabilities/**`                                |
| Adapter wiring                           | `apps/partner-ai-service/src/composition/service-composition.ts` and `src/adapters/**`   |
| Repository storage contracts/adapters    | `packages/db/src/schema-contract/**` and `packages/db/src/repositories/**`               |
| Prepared assistant turn execution        | `packages/agent-runtime/src/runtime/**`                                                  |
| Browser protocol and widget state        | `packages/chat-protocol`, `packages/chat-client`, `packages/side-chat-widget`            |

Short rule:

```txt
partner-ai-core owns portable product contracts, policy-facing declarations,
context ports, and what may become model-visible.

partner-ai-service owns SIDECHAT_* parsing, concrete deployment modes, adapter
selection, service composition, and safe diagnostics.
```

## Alignment Steps

1. Audit the phase 1 through 3 implementation against the ownership table.

   Confirm that core exports only portable capability contracts, manifest
   declarations, context ports, and policy-facing values. Confirm that service
   owns deployable modes such as no-op, Postgres, HTTP, external, and LangGraph.

2. Normalize config and status vocabulary.

   Service status may report adapter ids and concrete modes when they are safe
   diagnostics. Core manifest/config types must not gain adapter ids, env names,
   database modes, provider choices, or no-op deployment details.

3. Keep admission policy naming honest.

   If current service behavior still includes every gathered candidate, keep
   that fact visible in service diagnostics or implementation names. Do not make
   `deterministic_v1` imply budgeted trimming until the context-manager actually
   drops candidates under budget pressure.

4. Check context-port seams before adding behavior.

   Conversation history, memory, RAG, and research should enter model context
   through core-owned context ports or prepared-context contracts. Service
   adapters may retrieve from DB, HTTP, Postgres, static files, or external
   agents, but those implementation shapes must be adapted before crossing into
   core.

5. Correct remaining plan files before implementing them.

   Update later plans where they currently blur ownership. Required corrections:

   ```txt
   04: add a core-owned history context/read port decision before service DB wiring.
   04: keep message ids, sequence, and token estimates in manifest metadata, not runtime messages.
   05: name DB and persistent E2E verification commands.
   06: separate include-all current behavior from the future deterministic trimming policy.
   07: decide whether memory provenance/status fields are core port fields or DB metadata.
   08: remove adapter ids and service-only defaults from portable retrieval source declarations.
   09: align research modes with service config, or add any new mode deliberately before using it.
   10: update final definition of done after the corrected plans are precise.
   ```

6. Lock the alignment with tests or static checks where practical.

   Prefer existing service config, composition, health status, context-manager,
   and adoption-harness tests. Add targeted tests only where current coverage
   would allow an ownership regression to pass.

7. Keep docs and comments synchronized with the corrected boundary.

   Update plan docs, local READMEs, and boundary comments in the same patch if
   the implementation or plan wording changes package ownership, lifecycle
   order, manifest shape, or diagnostics semantics.

## Non-Goals

```txt
[ ] Do not implement conversation history context in this phase.
[ ] Do not add real Postgres persistence behavior beyond alignment tests.
[ ] Do not add durable memory recall/write behavior.
[ ] Do not add a real RAG retriever.
[ ] Do not add a research agent.
[ ] Do not change browser protocol or widget state unless a test fixture is wrong.
```

## Required Assertions

```txt
[ ] Core does not export service-only deployment modes or adapter ids.
[ ] Service config maps SIDECHAT_* values into core contracts plus service-only modes.
[ ] Production-like enabled memory/RAG/research cannot silently resolve to no-op.
[ ] Capability diagnostics expose safe status, not credentials, DB rows, retrieved content, or memory content.
[ ] Context admission diagnostics distinguish recorded budget from actual trimming behavior.
[ ] Remaining plan files no longer instruct future work to put core-owned contracts in service or service-owned wiring in core.
```

## Exit Criteria

```txt
[ ] Implemented phases 1 through 3 match the canonical package boundaries.
[ ] Later plans describe the corrected ownership split before they are executed.
[ ] The service can still report capability status without leaking implementation secrets.
[ ] Tests fail for silent no-op production-like capability enablement.
[ ] The next phase can add history context without importing DB rows, service modes, or adapter details into core.
```
