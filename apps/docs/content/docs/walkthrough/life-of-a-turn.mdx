---
title: Life of a Turn
description: One message — "show me my open Jira tickets" — traced from the browser to a durable, resumable answer.
---

Maya types "show me my open Jira tickets" and presses enter. What follows is **one turn**, and it
is the most important thing to understand in this codebase. If you picture it as a single request
that streams an answer back, you will misread half the system. A turn is actually **two HTTP calls
plus a server-owned fiber** — an Effect fiber is a lightweight thread, and this one outlives both
calls. That shape is what lets Maya reload the page mid-answer and rejoin the same turn.

<Callout title="You will learn">
- Why one turn is **two HTTP calls plus a server-owned fiber**, not a single streaming request.
- What the synchronous pre-start phase finishes before any model runs — and why its last act, the `started` event, is the fence.
- How generation stays pinned to one owner instance while any instance can serve the stream.
- Why the **durable log**, not the live connection, is the source of truth — so a reconnect is just a resubscribe.
- How `Effect.onExit` plus the reaper guarantee every turn reaches a terminal exactly once.
</Callout>

## Two calls, not one

The browser never holds the turn open. It makes two separate calls:

1. **`POST /chat/runs`** sends Maya's text, runs the synchronous setup, **forks generation onto a
   server-owned fiber**, and returns the turn's identity as JSON — not a stream.
2. **`GET /chat/turns/:assistantTurnId/stream`** opens the SSE connection that *watches* the turn.

The turn runs on the server regardless of whether anyone is watching. The browser is a subscriber,
not the owner. Every other guarantee in this section follows from that one decision.

## Pre-start: the synchronous spine

Steps 1–10 all run **inside `POST /chat/runs`**, in order, on the request. If any of them fail,
Maya gets a JSON error and no turn is created — nothing half-started leaks out.

Two of these steps carry most of the weight:

- **Start the assistant turn (step 6)** is the moment the turn becomes *durable*. The
  `assistant_turns` row is written as `running`, keyed idempotently on `requestId`, so a retried
  `POST` resolves to the same turn instead of starting a second one.
- **Append the user message (step 5)** is where the trust boundary bites. The server assigns
  `role: 'user'` to Maya's text itself; the browser never gets to set a role. The system prompt and
  the tool allowlist were chosen by the profile in step 3, not by her request.

Pre-start ends by emitting the `started` event (sequence 0), forking generation, and returning the
identity. That `started` event is the fence: before it, failures are synchronous JSON errors; after
it, the turn is live and every outcome is a durable protocol event.

<Callout type="warn" title="Pitfall">
Don't treat the `started` event as cosmetic. It is the boundary between two failure regimes: a
failure in steps 1–8 is a plain JSON error and **no turn exists**, while a failure after it is a
durable terminal event the stream will replay. Code that "handles a turn error" by watching the
`POST` response will miss every real generation failure — those land in the log, not the HTTP reply.
</Callout>

## The hand-off: why a fiber

Step 9 is the hinge. `FiberMap.run(fibers, assistantTurnId, …)` detaches generation onto a fiber
keyed by the turn's id, running in the **service scope** rather than the request scope. The `POST`
then returns immediately with `{ assistantTurnId, conversationId, requestId, status: 'running' }`.

Because generation lives on that fiber and not inside the HTTP response, Maya's connection is now
irrelevant to whether the answer gets produced. She can close the panel, reload, or lose her
network — the model keeps generating and the results keep landing in the durable log.

## Generation: on the fiber, owned by one instance

The fiber does four things, repeatedly, until the model is done:

- **Acquire an owner lease, then heartbeat it (step 11).** A compare-and-swap claims the turn for
  this server instance and stamps a lease epoch; a heartbeat renews it. If the reaper or a new owner
  bumps the epoch, the heartbeat self-interrupts — so exactly one instance ever writes a turn's
  events, even across restarts and deploys.
- **Run the tool loop (step 12).** The runtime streams `RuntimeEvent`s from the provider: text
  deltas, reasoning, and the `jira.search_issues` tool call, whose result is fed back to the model.
- **Map each event to the protocol (step 13).** A `RuntimeEvent` becomes a `sidechat.v1` event with
  the next sequence number. A small state machine rejects illegal transitions, so a turn can never
  emit two terminals or an event after one.
- **Append to the durable log (step 14).** Every event is written to `turn_events`, then
  `pg_notify` wakes any listeners. The log is the source of truth; the live notification is only a
  latency optimization.

## Stream: replay, then tail

`GET /chat/turns/:assistantTurnId/stream?after=<seq>` (step 15) serves the answer to the browser,
and it is built for reconnection:

- It **replays** every `turn_events` row past `after`, so a fresh subscriber catches up from the
  beginning (`after=-1`) and a reconnecting one resumes from its last seen sequence.
- It then **tails** live events — merging the dispatcher's fan-out with a periodic safety poll — and
  **ends at the first terminal event**.
- If the log was pruned past `after`, the route answers `replay_expired`, and the client falls back
  to reading conversation history instead.

Reconnecting is therefore not a special case. The browser just opens the same `GET` again with
`after` set to the last sequence it saw.

## Finalize: the turn always ends

Generation is wrapped in `Effect.onExit` (step 16), so the turn reaches a terminal **no matter how
the fiber exits**. The `onExit` handler:

- On success, writes `completed` with the assistant message and token usage.
- On interrupt or provider failure, appends a **conflict-free synthetic terminal** (a partial unique
  index guarantees exactly one terminal per turn) and records the honest status: `user_aborted` for a
  requested cancel, otherwise `provider_failed`.

Three background mechanisms back this up:

- **The reaper** sweeps for turns whose lease expired — a crashed owner — and terminalizes them, so
  no turn is stuck `running` forever.
- **The pruner** deletes old `turn_events` rows past the retention window; the turn record and its
  final message stay.
- **Cancellation** is a durable intent plus a notify: `POST …/cancel` flags the turn and signals the
  owning instance to interrupt the fiber, which finalizes as `user_aborted`.

<Callout title="What survives a disconnect">
Because the turn is a server-owned fiber writing to a durable log, and it always finalizes through
`onExit` with the reaper as a backstop, every interruption has a defined outcome: a browser reload
or lost connection just re-subscribes with `after`; a server crash mid-turn is reaped and
terminalized; and the answer is produced exactly once regardless of who is watching.
</Callout>

<Callout title="Recap">
- One turn is **two HTTP calls plus a server-owned fiber**: `POST /chat/runs` sets up and forks, `GET …/stream` watches.
- Pre-start (steps 1–10) runs synchronously; its last act, the `started` event, fences JSON errors from durable protocol events.
- Generation is pinned to one owner instance by a lease, but the durable `turn_events` log — not the connection — is the source of truth.
- Every turn reaches a terminal exactly once through `Effect.onExit`, with the reaper backstopping a crashed owner.
</Callout>
