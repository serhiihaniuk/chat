Maya types "show me my open Jira tickets" and presses enter. One assistant turn begins. The browser
keeps one SSE response open, but the server owns generation independently of that connection.

<Callout title="You will learn">
- What must finish before the service opens SSE.
- How a server-owned fiber keeps generating after a subscriber disconnects.
- Where live events exist and where the final answer becomes durable.
- How same-instance resume and history fallback differ.

</Callout>

## The fifteen hops

| ---: | --- | --- | --- | --- |
| 01 | Pre-start | Service | Validate `POST /chat/runs` | Parse JSON, validate `ChatStreamRequest`, and load proven auth |
| 02 | Pre-start | Core | Prove authority | Confirm the subject may act in the workspace |
| 03 | Pre-start | Core | Record request received | Create correlation and observability before model work |
| 04 | Pre-start | Core | Resolve the turn plan | Select profile, model, reasoning, tools, executor, and instructions |
| 05 | Pre-start | Core | Run turn guards | Fail closed before private context or tools |
| 06 | Pre-start | Core/DB | Ensure the conversation | Load or create only a conversation the subject may access |
| 07 | Pre-start | Core/DB | Reject a concurrent turn | Return `conversation_busy` if this conversation already has a running assistant turn |
| 08 | Pre-start | Core/DB | Append the user message | Persist Maya's visible message with the server-assigned user role |
| 09 | Pre-start | Core/DB | Start the turn record | Insert `assistant_turns` as `running`, idempotent on `requestId` |
| 10 | Pre-start | Core | Prepare context | Assemble history, host context, tool context, and the manifest snapshot |
| 11 | Hand-off | Service | Fork and register | Fork generation, then register the fresh ID before the POST response subscribes |

## Pre-start is synchronous

Steps 1–10 run before the browser receives `sidechat.started`. A failure here returns JSON instead of
opening a partial stream. If a turn record was already inserted, Core marks it failed before returning
the setup error.

The trust boundary is visible in this phase. The browser sends Maya's text. The server assigns its
role, proves authority, selects instructions, and closes the tool allowlist before any provider call.

## The hand-off creates two independent lifetimes

After preparation, the turn runner forks `runTurnGeneration` onto a `FiberMap`. The route registers
the new `assistantTurnId` in its in-memory turn-event registry before the POST response subscribes
to that registry from sequence `-1` and becomes SSE.

The response and the generation fiber now have separate lifetimes. A browser disconnect removes a
subscriber. It does not abort the provider or cancel the turn. Only the cancel route records durable
intent and interrupts the owner.

## Generation writes live events to memory

The fiber acquires an owner lease before emitting `sidechat.started`. The Runtime then produces
deltas, reasoning, tool activities, and a terminal outcome. Core maps those `RuntimeEvent`s to the
public `sidechat.v1` vocabulary and assigns a fresh browser sequence.

Each public event lands in the owner's in-memory registry. The registry is ordered and terminal
guarded, but not durable. It wakes only subscribers on that service instance.

## The stream resumes only on the owner

The POST response is the primary stream. If that connection fails after the widget learns the turn
identity, it may call `GET /chat/turns/:assistantTurnId/stream?after=<lastSeenSequence>`.

- On the same instance, the route replays the buffered suffix and tails new events.
- On another instance while the turn runs, the route returns `409 stream_unavailable`. The widget
  polls durable status until the turn ends, then loads history.
- After a terminal buffer is swept, the route returns `404 replay_expired`. The widget loads history
  immediately.

The browser may lose the typing animation. It does not lose the durable final answer.

## Finalization always runs

`runTurnGeneration` owns finalization through `Effect.onExit`. A normal terminal event determines the
durable outcome. If the fiber exits without a terminal, finalization appends one synthetic terminal
to the registry and records an honest failure status. A terminal already emitted by the stream wins
over a late interrupt.

An owner lease heartbeat detects fencing. A per-instance reaper claims expired turns in disjoint
batches and terminalizes them after a hard crash. Postgres therefore converges to a final turn status
even though the live event buffer disappeared with the process.

<Callout title="Recap">
- `POST /chat/runs` performs setup and returns the primary SSE stream.
- Generation runs on a server-owned fiber and outlives a disconnected subscriber.
- Live events exist only in the owning instance's registry.
- Postgres stores the final conversation, messages, and turn outcome.
- Same-instance resume uses `after`; non-owner or swept streams converge through status and history.
</Callout>
