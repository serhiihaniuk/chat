## Meet Maya (we'll follow her the whole way)

Maya is a support agent. She has the company's web dashboard open, and in the corner there's a
chat panel that slid in from the side. She types:

> show me my open Jira tickets

A second later, a little row appears that says "Searching Jira…", flips to "Searched Jira
(3 results)", and then a written summary of her three tickets types itself out, word by word.

That tiny interaction touches every part of this system — the wire format, the rules engine, an
AI model, a tool call out to Jira, and a live stream back to her screen. We'll follow Maya's exact
request through every section of this guide. When an idea feels abstract, come back to her.

## What the product is

Side Chat is a self-contained AI chat assistant that a company embeds into its existing web app —
that side panel Maya is using. The code lives in a monorepo: one git repository holding about a
dozen small TypeScript packages, plus one deployable server. Each package has one job, which is
what makes the pile navigable once you know the map.

In one breath: the user types a message → the server runs one **turn** → an LLM (today, OpenAI)
streams an answer back, pausing to call tools like "search Jira" when it needs to → the panel
renders it live. Every concept in this guide hangs off that sentence.

## The 60-second mental model

Side Chat is four layers wrapped around a protected center, and one rule explains most of the
design: **dependencies point inward**. The center holds the rules; it knows nothing about OpenAI,
HTTP, or the database. Those live at the edges and can be swapped without touching the rules.

Read the layers from the outside in. Each one reaches the layer beneath it only through a typed
contract:

| Layer | Package | Owns | Talks down through |
| --- | --- | --- | --- |
| **Browser** | `side-chat-widget`, `host-bridge` | The side panel Maya types into | the HTTP API (`POST /chat/runs`, `GET …/stream`) |
| **Service** | `apps/partner-ai-service` | HTTP routes, composition, SSE transport, the server-owned turn runner | the Core stream-chat entry point |
| **Core — the rules** | `packages/partner-ai-core` | Policy, context, and lifecycle; imports no infrastructure | the `AiRuntimePort` contract |
| **Runtime — execution** | `packages/agent-runtime` | Provider calls, the tool loop, event mapping | the provider SDK and the DB repositories |

Two contract packages carry the pure types (no runtime code) across those boundaries: `chat-protocol` between the
browser and the service, and `ai-runtime-contract` between Core and the Runtime. Results flow back
out the same path — the Runtime emits events, Core maps them to protocol events, and the service
streams them to the browser — but **no inner layer ever imports an outer one**.

Two rules are worth burning in now, because together they explain most of the design:

<Callout title="The two golden rules">
1. **Dependencies point inward.** The Core (`partner-ai-core`) is the most isolated package in the
   repo — it imports no database, no HTTP framework, no AI SDK. Everything it needs from the
   outside world it asks for through interfaces.
2. **The server owns the prompt and the roles; the browser only sends text.** Maya's panel sends
   her words and nothing else — it cannot tell the AI how to behave or put words in its mouth.
   This "trust boundary" shapes the request format and half the turn pipeline.
</Callout>

One thing worth calling out: the browser makes **two** calls, not one. `POST /chat/runs` hands
over Maya's text and gets back the turn's identity; the server then runs the turn **on its own**,
writing every event to a durable log. The browser opens `GET …/stream` to watch that log live.
Because the turn lives on the server — not inside one HTTP response — Maya can close the panel,
reload, or drop her connection, then reconnect to the **same** turn still in progress. *Life of a
Turn* traces exactly how.

## Two TypeScript things that aren't AI, but will confuse you on day one

Defuse these now, before you open a file and panic. Neither is essential to understanding the
architecture — they're just unfamiliar machinery you'll see everywhere.

- **Effect.** This codebase uses the [Effect](https://effect.website) library instead of plain
  `Promise`. When you see `Effect.Effect<A, E>` or `Stream.Stream<A, E>`, read it as "a lazy,
  not-yet-run Promise of `A` that can fail with the typed error `E`." That's enough to navigate
  everything.
- **Branded types.** You'll see `RequestId = Brand<string, "RequestId">`. At runtime that is just
  a string. The `Brand` part is a compile-time-only label that stops you from passing a
  `ConversationId` where a `RequestId` belongs. Treat `RequestId` as "a string the compiler is
  being careful about" and move on.

With Maya in mind, the four layers mapped, the two golden rules set, and those two gotchas
defused, the rest of the guide has somewhere to stand. Next: [the words](/docs/walkthrough/vocabulary).
