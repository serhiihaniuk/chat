## Meet Maya

Maya is a support agent. She opens the Side Chat panel in her company's dashboard and types:

> show me my open Jira tickets

The panel shows a "Searching Jira…" activity, then streams a summary. That small interaction crosses
the browser protocol, product policy, a provider, a tool, persistence, and a live SSE response.

<Callout title="You will learn">
- The four layers and the rule that keeps them separate.
- Why the browser sends user text while the server owns trusted instructions and roles.
- Why generation outlives the stream connection.
- What connection-bound streaming preserves, and what it deliberately does not preserve.

</Callout>

## The four layers

Dependencies point inward. Each layer crosses the next boundary through a typed contract.

| Layer | Package | Owns | Talks down through |
| --- | --- | --- | --- |
| **Browser** | `side-chat-widget`, `host-bridge` | The panel, API client, run store, and host seam | `chat-protocol` |
| **Service** | `apps/partner-ai-service` | HTTP, composition, SSE, the turn runner, and live registry | Core entry points |
| **Core** | `packages/partner-ai-core` | Authority, policy, context, lifecycle, and protocol mapping | `ai-runtime-contract` |
| **Runtime** | `packages/agent-runtime` | Provider calls, tool loops, and runtime events | Provider SDKs |

The Core never imports HTTP, Postgres, React, or a provider SDK. The browser never receives a
provider DTO, database row, or `RuntimeEvent`.

## Two rules explain the design

<Callout title="The two golden rules">
1. **Dependencies point inward.** Outer layers choose adapters; inner layers depend on contracts.
2. **The server owns authority and instructions.** The browser contributes user text and optional
   catalog-backed preferences. It cannot assign roles or send provider-native options.
</Callout>

## One POST starts and streams the turn

The widget calls `POST /chat/runs`. The service performs all pre-start checks, forks generation onto
a server-owned Effect fiber, registers a live buffer on that instance before subscribing, and
returns SSE on the same response. The first event, `sidechat.started`, identifies the turn.

The browser is a subscriber, not the worker. Closing the stream releases that subscriber but does
not cancel generation. The fiber continues and Postgres records the final status and assistant
message.

The live events themselves are connection-bound. They exist in the owning instance's in-memory
turn-event registry. A reconnect can resume from the last sequence if it reaches the same instance.
If it reaches another instance, the server returns `409 stream_unavailable`; the widget polls status
and loads the final answer from history. If a terminal buffer was already swept, `404 replay_expired`
also sends the widget to history.

<Callout type="warn" title="Pitfall">
Server-owned generation does not mean cross-instance live replay. The final answer survives in
Postgres. The typing animation survives only while the owning process still holds its registry
entry.
</Callout>

## Two TypeScript ideas

- **Effect.** Read `Effect.Effect<A, E>` as a lazy asynchronous operation that succeeds with `A` or
  fails with typed error `E`. `Stream.Stream<A, E>` is the streaming counterpart.
- **Branded types.** `RequestId = Brand<string, "RequestId">` is a string with a compile-time label
  that prevents accidental id mixing.

<Callout title="Recap">
- Side Chat has Browser, Service, Core, and Runtime layers with dependencies pointing inward.
- The server owns trusted instructions, authority, generation, and final persistence.
- `POST /chat/runs` is the primary SSE stream.
- Same-instance resume uses `GET …/stream?after=<seq>`; other cases converge through status and history.
</Callout>
