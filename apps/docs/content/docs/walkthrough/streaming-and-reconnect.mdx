Maya closes her laptop while the answer is streaming. The browser connection disappears, but the
server-owned generation fiber keeps running. When she returns, the widget tries to recover the live
stream. If live recovery is impossible, it waits for the durable outcome and reloads the transcript.

<Callout title="You will learn">
- Why the POST response is the primary stream.
- How the owning instance buffers and replays live events.
- Why another instance returns `stream_unavailable` instead of an empty stream.
- How `replay_expired` and conversation history complete recovery.
- Which three cross-instance signals still use Postgres notifications.

</Callout>

## The live registry is a process-local notebook

Picture a numbered notebook that exists only in one server process. Every mapped `sidechat.v1` event
is one line: `sidechat.started` at sequence 0, then deltas and activities, then exactly one terminal.
The registry keeps those lines ordered and wakes local subscribers.

The notebook is useful but disposable. It is not stored in Postgres and vanishes when the process
restarts. Postgres stores the settled conversation, messages, turn record, usage, and cancel intent.
That final state is what the product promises to preserve.

## The POST response opens the notebook

`POST /chat/runs` performs setup before any SSE frame. After setup succeeds, the service forks
generation, registers the turn in its local notebook before subscribing, and streams the same
response from the start. The first frame identifies the turn. No separate GET is required for the
happy path.

Closing the POST response closes one window onto the notebook. The worker remains alive. To stop the
turn, the widget must call `POST /chat/turns/:assistantTurnId/cancel`.

## Same-instance resume uses `after`

The widget tracks the last sequence it applied. A recovery request uses:

`GET /chat/turns/:assistantTurnId/stream?after=<lastSeenSequence>`

On the owning instance, the server registers the subscriber first, replays buffered events with a
higher sequence, then tails new events. Registering first prevents a gap between replay and tail.
A dense sequence gate catches a dropped queue wake-up by re-reading the buffer.

This is exact while the local registry entry exists. It is not a cross-instance contract.

## Another instance fails fast

A load balancer may send the resume GET to a different service instance. That process can read the
durable turn row, but it cannot read the owner's live registry. If the turn is still `running`, the
route returns HTTP `409 stream_unavailable` with reason `not_stream_owner`.

The widget then polls `GET /chat/turns/:assistantTurnId`. When the durable status becomes terminal,
it refetches conversation history and replaces the transient run state with the saved answer.

Failing fast matters. Opening SSE from a non-owner would produce an empty connection that could never
deliver a terminal event.

## A swept terminal buffer returns `replay_expired`

The registry lazily sweeps terminal entries after their retention window and after all subscribers
leave. Sweeping frees process memory; it does not delete the turn or assistant message.

If a client asks for a terminal turn after its buffer is gone, the stream route returns HTTP
`404 replay_expired` before opening SSE. The widget stops retrying the live stream, reloads
conversation history, and clears the persisted run marker.

<Callout title="What survives">
- A network disconnect: generation and the owner's registry entry continue.
- A same-instance reconnect: the buffered suffix replays from `after`.
- A cross-instance reconnect: live animation may be lost; status and history recover the answer.
- A process crash: the registry is lost; the reaper settles the durable turn record.
</Callout>

## Cross-instance signals are separate from turn content

Postgres `LISTEN/NOTIFY` does not carry turn deltas. It remains useful for three coordination paths:

| Signal | Why another instance needs it | Durable recovery |
| --- | --- | --- |
| Cancel | Interrupt the instance that owns the generation fiber | Re-scan running turns with cancel intent |
| Turn activity | Update generating indicators across tabs and instances | Re-read the subject-scoped running-turn snapshot |
| Host-command result | Settle a host command waiting on the owner | Poll the stored command result |

Each notification is a wake-up, not trusted state. Each listener reconnects after a dropped Postgres
connection and recovers from the durable source named above.

`GET /chat/activity` uses the turn-activity signal to power generating indicators. It is a separate
subject-scoped SSE stream with a snapshot and live transitions. It has no turn sequence, replay
cursor, or terminal event.

<Callout title="Recap">
- The POST response is the primary SSE stream.
- The owner stores live events in an in-memory registry.
- Same-instance resume replays `sequence > after`, then tails.
- `409 stream_unavailable` means poll status; `404 replay_expired` means read history.
- Cancel, turn activity, and host-command results use Postgres notifications. Turn content does not.
</Callout>
