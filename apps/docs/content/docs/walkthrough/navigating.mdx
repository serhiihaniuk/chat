Use this page when you arrive with a concrete question. It points to the smallest set of files that
explains the current implementation.

## Where is the system prompt built?

The service config declares the prompt sections. The prompt builder joins them. Core's
`buildModelMessages` places the resolved system instructions before admitted context, history, and
the current user message. The browser cannot supply or reorder those roles.

Start in these files:

- `apps/partner-ai-service/src/composition/assistant/default-assistant-config.ts`
- `apps/partner-ai-service/src/composition/assistant/system-prompt-builder.ts`
- `packages/partner-ai-core/src/application/stream-chat/model-request/build-model-turn-request.ts`

## Where are tools registered?

`apps/partner-ai-service/src/composition/tools/service-tool-registry.ts` binds each declared
`ToolCapability` to a matching executable `RuntimeTool`. It rejects name mismatches and duplicates
during composition. Concrete tool adapters live under
`apps/partner-ai-service/src/adapters/tools/`.

## Where does a turn start?

The primary route is `POST /chat/runs` in
`apps/partner-ai-service/src/inbound/http/routes/chat/runs/chat-runs.ts`. It validates the request,
calls the turn runner, then returns the turn's SSE stream on the same response.

`inbound/turn-runner/turn-runner.ts` runs `prepareStreamChatTurn`, registers only newly inserted turns,
and forks `runTurnGeneration` onto a server-owned fiber. The first streamed event,
`sidechat.started`, carries the turn identity.

`GET /chat/turns/:assistantTurnId/stream?after=<seq>` in `chat-turns.ts` is the same-instance resume
route. It never starts generation. A non-owner running turn returns `409 stream_unavailable`; a
terminal turn without a registry buffer returns `404 replay_expired`.

## Recommended reading order

1. Read `packages/chat-protocol/src/sidechat-v1/events/event-union.ts` for the browser contract.
2. Read `prepare-stream-chat-turn.ts` and `run-turn-generation.ts` for the lifecycle split.
3. Read `turn-runner.ts`, `chat-runs.ts`, and `turn-subscription-stream.ts` for service ownership.
4. Read `in-memory-turn-event-log.ts` for the live registry rules.
5. Read `packages/agent-runtime/src/runtime/ai-sdk/` last for provider execution.

## Path-to-layer heuristic

| Path fragment | Layer | What belongs there |
| --- | --- | --- |
| `partner-ai-core/src/domain/` | Domain | Pure types and validation |
| `partner-ai-core/src/ports/` | Ports | Interfaces describing what Core needs |
| `partner-ai-core/src/application/` | Use cases | Effect orchestration and protocol mapping |
| `partner-ai-service/src/adapters/` | Adapters | Concrete persistence, tools, auth, and observability |
| `partner-ai-service/src/composition/` | Composition | Registry validation and adapter wiring |
| `partner-ai-service/src/inbound/` | Transport | Routes, SSE framing, runner, and subscription streams |
| `side-chat-widget/src/` | Browser | React UI, API client, run state, and protocol projection |

Dependencies point inward. Core never imports an adapter. The widget never imports service, runtime,
Effect, or database code.

## Current edges worth knowing

- OpenAI is the shipped real provider; tests also use a fake provider.
- Turn guards are an adopter seam. The local/default composition may intentionally register none.
- Live stream resume is connection-bound. Do not assume another instance can replay a running turn.
- `sidechat.blocked` is a distinct terminal event, not an error alias.

## Tests that pin the contracts

- `chat-protocol` codec and ordering tests pin SSE frame identity, sequence, and terminal rules.
- `partner-ai-core` stream-chat tests pin pre-start ordering and fail-closed guards.
- Service route tests pin POST-as-SSE, non-owner `stream_unavailable`, and terminal
  `replay_expired` behavior.
- The in-memory registry tests pin sequence conflicts, gaps, terminal closure, and shutdown.
- Widget SSE and run-controller tests pin decode, reconnect, status polling, and history fallback.
- Composition registry tests pin duplicate declarations and name mismatches at boot.

Start a change by reading the nearest test, then the production file it protects.
