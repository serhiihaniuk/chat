This is the payoff. "Tool flow" sounds like magic; it is not. A tool is two plain objects tied
together by a shared name, registered once, then checked by several independent gates before the
model can touch it. Let's build the real `jira.search_issues` tool that answers Maya's "show me my
open Jira tickets" and follow her request through every gate.

<Callout title="You will learn">
- Why every tool is split into two halves — the **capability** the model reads and the **executable** the server runs.
- How a shared `name` constant ties the halves together, and why the registry checks it at boot.
- How the trusted `toolScope` (Maya's `subjectId`) reaches `execute` without the model ever supplying it.
- The four gates a tool name passes before the model can call it.
- The one shape — four small objects — that every new tool repeats.
</Callout>

## The idea

A tool is a function you let the model call. But the model is not trusted, so you split each tool
into two halves:

- **The menu card the model reads** — `name`, `description`, a JSON schema of inputs. The model uses
  this to decide whether to call it. It never sees the code.
- **The recipe card the server runs** — the actual `execute` function. The model never sees this
  either.

A diner reads a menu (the capability) and orders a dish by name. The kitchen has a recipe (the
executable) for that exact name. A head waiter checks tonight's order ticket (the turn policy) to
confirm the diner is allowed that dish. Three different people, one dish name tying them together.
You physically cannot declare a dish with no recipe behind it, and the model can only order from the
ticket it was handed.

## Step 1 — Write the executable (RuntimeTool)

The executable is a `RuntimeTool`: a name, description, input schema, and an `execute` function.
`execute` receives the input the model produced (a plain `JsonObject`) and a `RuntimeToolContext`
carrying the trusted server-side scope. From the real `adapters/tools/examples/jira-search-issues-tool.ts`:

```ts
export const createJiraSearchIssuesTool = ({ jiraClient }): RuntimeTool => ({
  name: JIRA_SEARCH_ISSUES_TOOL_NAME, // "jira.search_issues"
  description: "Search Jira issues the current user may access...",
  inputSchema: JIRA_SEARCH_ISSUES_INPUT_SCHEMA,
  readSources: readJiraIssueSources,
  execute: (input, context) =>
    Effect.gen(function* () {
      const searchInput = yield* readJiraSearchIssuesInput(input); // { query, maxResults }
      const scope = yield* readJiraToolScope(context);             // requires context.scope
      const issues = yield* jiraClient.searchIssues({
        query: searchInput.query,
        maxResults: searchInput.maxResults,
        subjectId: scope.subjectId, // Maya's id — Jira enforces per-user visibility
        workspaceId: scope.workspaceId,
        // ...requestId, hostAppId, conversationId, assistantTurnId, abortSignal
      }).pipe(Effect.mapError(toJiraToolError));
      return toJiraSearchIssuesResult(issues); // normalized JsonObject
    }),
});
```

Two things are the crux of why tools are safe:

- **What `execute` receives.** The model only ever sends `{ query, maxResults }`. Everything that
  decides *whose* tickets — `hostAppId`, `workspaceId`, `subjectId` — comes from `context.scope`,
  which the server injected. A hostile prompt has no field to ask for another user's tickets;
  `subjectId` is Maya's, full stop. `readJiraToolScope` fails closed: if the scope is missing, the
  tool errors with `TOOL_FAILED` rather than calling Jira with undefined ids.
- **What `execute` returns.** Not raw Jira JSON — a normalized result. `toJiraSearchIssuesResult`
  flattens each issue to `{ key, summary, status, url }`.

<Callout title="Deep dive">
`readSources` is optional: on the completed result it produces protocol-safe `{ label, url }`
metadata that the widget renders as clickable links. The runtime renders no UI — it only attaches
data. You can skip `readSources` entirely and the tool still works; it just won't surface source
links in the panel.
</Callout>

## Step 2 — Write the capability (ToolCapability)

The capability is what the model sees. Three fields, no code:

```ts
export const JIRA_SEARCH_ISSUES_INPUT_SCHEMA = {
  type: "object",
  properties: {
    query:      { type: "string",  description: "Jira issue search text or JQL fragment." },
    maxResults: { type: "integer", minimum: 1, maximum: 25, description: "Maximum issues to return." },
  },
  required: ["query"],
  additionalProperties: false,
} as const satisfies JsonObject;
```

No `subjectId` or `workspaceId` in the schema — those are server concerns, deliberately invisible to
the model. The model can only produce `query` and `maxResults`. The `minimum`/`maximum` is
belt-and-suspenders: the runtime clamps server-side, so even `maxResults: 9999` quietly falls back to
a safe cap. "The model is not trusted" made concrete.

## Step 3 — Pair them (createServiceToolRegistration)

You never register the two halves separately. You bundle them in one call, and the registration name
is taken from `runtimeTool.name`:

```ts
export const createJiraSearchIssuesRegistration = ({ jiraClient }): ServiceToolRegistration =>
  createServiceToolRegistration({
    capability: {
      name: JIRA_SEARCH_ISSUES_TOOL_NAME,
      description: "Search Jira issues visible to the current user.",
      inputSchema: JIRA_SEARCH_ISSUES_INPUT_SCHEMA,
    },
    runtimeTool: createJiraSearchIssuesTool({ jiraClient }),
  });
```

The menu card and the recipe card travel together in one object. You cannot create the registration
with only one half.

## Step 4 — Register and split (createServiceToolRegistry)

At startup, `createServiceToolBundle` hands all registrations to `createServiceToolRegistry`, which
validates, then splits:

```ts
export const createServiceToolRegistry = (registrations) => {
  const seenNames = new Set<string>();
  for (const registration of registrations) {
    assertMatchingNames(registration); // capability.name === runtimeTool.name === registration.name
    assertUniqueName(seenNames, registration.name);
  }
  return {
    toolCapabilities: registrations.map((r) => r.capability), // → the manifest (the model's menu)
    runtimeTools:     registrations.map((r) => r.runtimeTool), // → agent-runtime (the executables)
    // ...
  };
};
```

`assertMatchingNames` is the first gate, and it fires at boot. Rename the capability to `jira.search`
but forget the runtime tool, and the process throws before the HTTP server accepts a connection.

<Callout type="warn" title="Pitfall">
Resist the urge to dispatch tools with a hand-written `switch (toolName)`. There, a renamed or
mistyped tool name compiles cleanly and surfaces only when the model calls the bad name on a real
request — and then silently returns `undefined`. Pairing the two halves in one registration turns
that latent bug into a **boot-time crash with the exact offending name**, before any user is served.
</Callout>

## Step 5 — At turn time: a name-only allowlist crosses the boundary

Maya hits send. Core loads the manifest, resolves the turn policy, and validates the decision's
`allowedToolNames` against several independent checks. Here are the gates on our tool, concretely:

| Gate | Where | What it proves |
| --- | --- | --- |
| 1. Boot-time name match | `assertMatchingNames` | The capability and the executable share one name; no phantom tools |
| 2. Per-turn manifest check | `manifestToolIssues` | The tool is declared in this deployment's manifest |
| 3. Per-turn profile check | `profileToolIssues` | The tool is in this assistant profile's allowlist |
| 4. Approval policy | `approvalRequirementIssues` | Any required human approval is present |

`buildModelTurnRequest` then maps the validated decision into the request the runtime receives. The
critical field is `toolNames` — names only, no schemas, no executors — and the trusted `toolScope`:

```ts
export const buildModelTurnRequest = (input, turn): AiRuntimeRequest => ({
  // ...
  toolNames: turn.policyDecision.allowedToolNames, // ['jira.search_issues']
  toolScope: {
    hostAppId: input.hostAppId,
    workspaceId: turn.authContext.workspaceId,
    subjectId: turn.authContext.subject.subjectId, // Maya — injected here, trusted
    conversationId: turn.conversation.conversationId,
    assistantTurnId: turn.assistantTurnId,
    allowedHostCommandNames: turn.policyDecision.allowedCommandNames,
  },
  // ...
});
```

This is the architectural seam: before this line a tool is just a string; after it, `agent-runtime`
resolves it to an executable.

That `toolScope` is the other end of the loop from step 1. The AI SDK adapter sets
`scope: request.toolScope` when it wraps each tool, and `readJiraToolScope` reads it back — so the
executable learns it is running for Maya without the model ever supplying her id. Inside the runtime,
`selectRuntimeToolsByName(registry, names)` turns the names back into executables, throwing
`TOOL_UNAVAILABLE` on a miss.

## Step 6 — The loop: call, execute, activity, answer

Now the model runs. The AI SDK `ToolLoopAgent` streams parts; each maps to a `RuntimeEvent`, which
maps to a browser-visible `sidechat.activity` or `sidechat.delta`:

| Model / AI SDK | Runtime event | Browser event |
| --- | --- | --- |
| emits `tool-input-start` | `runtime.activity` (running, empty input) | `sidechat.activity` — "Searching Jira" |
| emits `tool-call` | `runtime.activity` (running, with query) | `sidechat.activity` — same row, input filled |
| invokes `execute` | `jiraClient.searchIssues({ subjectId: Maya })` → result | — |
| receives `tool-result` | `runtime.activity` (completed, result + sources) | `sidechat.activity` — same row, done |
| writes its answer | `runtime.output_delta` ×N | `sidechat.delta` — "Here are your open tickets…" |
| stream ends | `runtime.completed` | `sidechat.completed` |

All three tool events share one `activityId` (the `toolCallId`), so Maya sees a single "Searching
Jira" row update in place — running → running with the query → completed with clickable tickets —
rather than three rows flickering by. `readSources` fires on the result part; `mapToolResult` copies
the `{ label, url }` metadata onto `details.tool.sources`, and the panel renders them as links.

## The same shape, any tool

Nothing above was Jira-specific except the body of `execute`. To add a Slack search, a calendar
lookup, or a database query, you write the same four things:

1. A `RuntimeTool` with an `execute(input, context)`.
2. A `ToolCapability` with a matching `name` + `inputSchema`.
3. One `createServiceToolRegistration({ capability, runtimeTool })`.
4. Drop it into the bundle.

The manifest, policy, request builder, registry, and stream loop do not change. They are generic over
"a tool."

## The safety guarantees, restated

The name `jira.search_issues` lives in several places and they cannot drift:

- The `JIRA_SEARCH_ISSUES_TOOL_NAME` constant is the source of truth.
- `ToolCapability.name` (manifest) and `RuntimeTool.name` (registry) are checked equal by
  `assertMatchingNames` at boot (gate 1).
- `TurnPolicyDecision.allowedToolNames` is checked against both the manifest and the profile per turn
  (gates 2 and 3).
- The AI SDK tool set is keyed from that validated allowlist.

Three guarantees fall out:

- **You cannot declare a tool with no executable.** Capability and `RuntimeTool` are born in one
  `ServiceToolRegistration`; the registry rejects a mismatch at boot.
- **Names are validated before the server starts.** A typo is a startup crash with the exact bad
  name, not a silent mid-stream failure.
- **The runtime only ever receives allowed names.** Core sends a closed allowlist checked against both
  the deployment's manifest and the active profile, so a tool Maya's workspace never enabled cannot
  reach the model — even if the model would love to call it.

<Callout title="Recap">
- A tool is two halves — the **capability** (what the model reads) and the **executable** (what the server runs) — joined by one shared `name`.
- You always pair them in a single `createServiceToolRegistration`; the registry rejects a mismatch at boot.
- The trusted `toolScope` carries Maya's `subjectId` into `execute`, so the model can never ask for another user's data.
- Four independent gates — name match, manifest, profile, approval — must pass before the model receives the name.
- Every new tool repeats the same four objects; the manifest, policy, request builder, and stream loop stay generic.
</Callout>
