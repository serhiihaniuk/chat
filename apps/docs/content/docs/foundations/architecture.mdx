The widget is defined entirely in tokens. Components never hard-code a colour, radius, or
spacing value — they read a token, and tokens cascade down three tiers.

## The three tiers

- **Tier 1 — Primitive.** Raw palette + scales: `--background`, `--foreground`, `--primary`,
  `--radius`, the `--text-*` and size scales. Declared on `:root`, overridden by the host's
  `.dark` and by named themes (`data-sidechat-theme`).
- **Tier 2 — Component.** Every part owns named variables that are *aliases* of tier 1 —
  e.g. `--switch-track-on: var(--primary)`. Declared on `.side-chat-widget-root` so theme
  overrides re-substitute them.
- **Tier 3 — Hook classes.** Small classes like `.sc-switch-root` read tier-2 tokens. This is
  the only layer that touches real CSS properties.

## Why tier 2 lives on the widget root

If `--switch-track-on: var(--primary)` were declared on `:root`, it would freeze to the
graphite `--primary` and inherit that frozen value down — a `data-sidechat-theme="sage"` on the
widget root could never re-skin it. Declaring tier-2 on the *same element* the theme attribute
targets means `var(--primary)` re-substitutes the themed value. This is the re-skin contract.

## Rules every component follows

- No arbitrary `[…]` values and no literal colours — only ledger utilities and tokens.
- Base UI state via named variants (`checked:`, `highlighted:`, `pressed:`), never `:hover` /
  `data-[…]:` on a Base UI part.
- Popups portal into the widget root via `usePortalContainer()` so they keep the theme + font.
