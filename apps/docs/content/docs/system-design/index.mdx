## The shape: four layers, dependencies inward

Side Chat is four layers wrapped around a protected center. Each layer depends only on the layer
beneath it, and only through a typed contract. The center stays pure: it never imports HTTP, a
database, or an AI SDK.

| Layer | Package | Owns |
| --- | --- | --- |
| Browser | `side-chat-widget`, `host-bridge` | The embedded panel, the API client, and the host bridge |
| Service | `apps/partner-ai-service` | HTTP routes, env/config, SSE transport, and the server-owned turn runner |
| Core | `packages/partner-ai-core` | Policy, context, capability contracts, lifecycle, and protocol mapping |
| Runtime | `packages/agent-runtime` | Provider calls, the tool loop, and runtime-event mapping |

Two contract packages hold the pure types that cross those boundaries: `chat-protocol`
(browser ⇄ service) and `ai-runtime-contract` (core ⇄ runtime).

## Streaming is resumable and server-owned

A turn does not live inside one HTTP response. It runs in two calls, so generation outlives any
single connection:

1. `POST /chat/runs` runs the pre-start steps, then returns the turn's identity as JSON —
   `{ protocolVersion, requestId, assistantTurnId, conversationId, status }`. The service forks
   generation onto a server-owned fiber.
2. `GET /chat/turns/:assistantTurnId/stream?after=<seq>` opens an SSE stream that replays the
   durable event log from `after`, then tails live events to the terminal one.

The durable `turn_events` log is the source of truth; the browser is only a subscriber, so a
reconnect resumes the same turn. A separate `GET /chat/activity` stream pushes turn lifecycle
across conversations, so the sidebar shows a live "generating" dot even on chats that are not open.

## What each package owns

| Package | Owns | First files to open |
| --- | --- | --- |
| `apps/partner-ai-service` | HTTP routes, composition, SSE transport, the turn runner, the dispatchers, the reaper and pruner | `src/inbound/http/app.ts`, `src/inbound/turn-runner/turn-runner.ts` |
| `packages/partner-ai-core` | Stream-chat workflow, policy, context, ports, lifecycle, protocol mapping | `src/application/stream-chat/` |
| `packages/ai-runtime-contract` | Provider-neutral request, tool scope, `RuntimeEvent`, error, stream, and port contracts | `src/index.ts` |
| `packages/agent-runtime` | Prepared-turn execution, executors, runtime tools, the provider adapter | `src/runtime/agent-runtime.ts` |
| `packages/chat-protocol` | `sidechat.v1` request/event DTOs, validators, the SSE codec, the generated schema | `src/sidechat-v1/index.ts` |
| `packages/side-chat-widget` | The React widget, the API client and SSE reader, the query repository, protocol-to-UI state | `src/widgets/side-chat/` |
| `packages/host-bridge` | The browser seam for host context and host commands | `src/bridge/` |
| `packages/db` | Persistence schema, repository contracts and adapters, the `LISTEN/NOTIFY` source | `src/schema-contract/`, `src/repositories/` |

## Invariants

These hold across the system; breaking one is a design bug, not a style choice:

- Product policy, capability configuration, and prepared context stay in `partner-ai-core`.
- Shared core-to-runtime request and event shapes stay in `ai-runtime-contract`.
- Provider and AI SDK details stay in `agent-runtime`.
- Browser contracts stay in `chat-protocol`, `host-bridge`, and the widget package.
- Env parsing and concrete adapter modes live in service composition, injected through ports.
- The repo ships no production host app.
