## Four layers, dependencies inward

Each layer depends only on the contract beside it. The Core stays free of HTTP, database drivers,
and provider SDKs.

| Layer | Package | Owns |
| --- | --- | --- |
| Browser | `side-chat-widget`, `host-bridge` | The embedded panel, API client, live-stream reader, and host seam |
| Service | `apps/partner-ai-service` | HTTP routes, SSE transport, the turn runner, and the in-memory event registry |
| Core | `packages/partner-ai-core` | Policy, context, capability contracts, lifecycle, and protocol mapping |
| Runtime | `packages/agent-runtime` | Provider calls, the tool loop, and runtime-event mapping |

Two pure contract packages cross those boundaries: `chat-protocol` between browser and service, and
`ai-runtime-contract` between Core and Runtime.

## Streaming is server-owned and connection-bound

`POST /chat/runs` performs pre-start work, forks generation onto a server-owned fiber, registers
the live turn on that service instance before subscribing, and returns the turn's SSE stream on the
same response.
`sidechat.started` is sequence 0 and carries the turn identity.

Live events exist only in that instance's in-memory turn-event registry. Postgres stores the durable
conversation, messages, turn status, and final answer. The browser can use
`GET /chat/turns/:assistantTurnId/stream?after=<seq>` to resume from the registry on the same
instance. A running turn on another instance returns `409 stream_unavailable`; the widget polls its
status and reads history after it becomes terminal. A terminal turn whose buffer was swept returns
`404 replay_expired`, which also sends the widget to conversation history.

`GET /chat/activity` is a separate subject-scoped SSE stream. It pushes cross-conversation lifecycle
signals for the sidebar's generating indicator; it is not a replay path for turn content.

## What each package owns

| Package | Owns | First files to open |
| --- | --- | --- |
| `apps/partner-ai-service` | HTTP routes, composition, SSE, turn runner, in-memory registry, reaper, and cross-instance dispatchers | `src/inbound/http/app.ts`, `src/inbound/turn-runner/turn-runner.ts` |
| `packages/partner-ai-core` | Stream-chat workflow, policy, context, ports, lifecycle, protocol mapping | `src/application/stream-chat/` |
| `packages/ai-runtime-contract` | Provider-neutral request, `RuntimeEvent`, error, stream, and port contracts | `src/index.ts` |
| `packages/agent-runtime` | Prepared-turn execution, executors, runtime tools, and provider adapters | `src/runtime/agent-runtime.ts` |
| `packages/chat-protocol` | `sidechat.v1` request/event DTOs, validators, SSE codecs, and schema | `src/sidechat-v1/index.ts` |
| `packages/side-chat-widget` | React widget, browser API client, SSE reader, query repository, and protocol-to-UI state | `src/widgets/side-chat/` |
| `packages/host-bridge` | Browser seam for host context and host commands | `src/bridge/` |
| `packages/db` | Persistence plus cancel, turn-activity, and host-command-result `LISTEN/NOTIFY` sources | `src/schema-contract/`, `src/repositories/` |

## Invariants

- Product policy, capability configuration, and prepared context stay in `partner-ai-core`.
- Provider and AI SDK details stay in `agent-runtime`.
- Browser contracts stay in `chat-protocol`, `host-bridge`, and the widget.
- Live turn events never become database rows; only final state is durable.
- Closing an SSE connection releases a subscriber. It never cancels generation.
- The repo ships no production host app.
