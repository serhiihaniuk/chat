This page explains the live transport between Core and the browser. For event meanings, read
[Protocol & Runtime Events](/docs/system-design/protocol-and-events). For lifecycle order, read
[Assistant Turn Lifecycle](/docs/system-design/assistant-turn).

## The live source is an in-memory registry

Every service instance owns one turn-event registry. After generation is forked, the route registers
the fresh turn before the POST response subscribes. Core maps each `RuntimeEvent` to `sidechat.v1`;
the owner appends that public event to the registry and wakes subscribers on the same instance.

The registry enforces the live-stream invariants:

- Events are ordered by a strictly increasing `sequence`.
- Re-appending an identical sequence is idempotent.
- Conflicting content at one sequence fails loudly.
- A gap is not emitted until the missing sequence reaches the buffer.
- The first terminal closes the turn; later appends are ignored.

Postgres never stores these frames. It stores the durable final turn and assistant message.

## Primary stream: the POST response

`POST /chat/runs` performs pre-start work before opening a stream. When setup succeeds, the service:

1. Registers the fresh `assistantTurnId` in the local registry.
2. Forks generation onto the instance's `TurnRunner`.
3. Subscribes from `after=-1`.
4. Returns that subscription as the POST response body with SSE headers.

`sidechat.started` is sequence 0, so the POST caller learns `assistantTurnId` from the event envelope
and `conversationId` from the event. Closing the response drops only that subscriber; generation
continues on the server-owned fiber.

## Resume: replay, then tail

`GET /chat/turns/:assistantTurnId/stream?after=<seq>` is the resume route. On the owning instance it
registers the subscriber first, replays buffered events with `sequence > after`, then tails local
notifications until a terminal event.

Each subscriber has a bounded dropping queue. A low-frequency safety poll re-reads the registry, and
the dense sequence gate detects a gap and catches up from the buffer. A dropped wake-up can delay an
event but cannot permanently lose it while the registry entry exists.

## Instance ownership is visible at the API

The live buffer exists only on the instance that started the turn. The route checks the registry
before writing any SSE frame:

- A running turn absent from this instance returns HTTP `409 stream_unavailable` with reason
  `not_stream_owner`. The widget polls durable status until terminal, then reads history.
- A terminal turn absent from this instance returns HTTP `404 replay_expired`. The widget reads the
  final answer from conversation history.
- A terminal turn still in the registry replays through its terminal and closes immediately.

This behavior prevents an empty non-owner stream from hanging. Same-instance resume is exact;
cross-instance live replay is deliberately unavailable.

## Registry sweep

The registry drops terminal entries only after they have no subscribers and the terminal retention
window has elapsed. The sweep is lazy and runs when the instance registers another turn. Sweeping
reclaims process memory; it does not delete any durable conversation or message.

## What still uses Postgres notifications

Turn content does not use `LISTEN/NOTIFY`. Three separate cross-instance coordination paths do:

| Channel | Announces | Recovery source |
| --- | --- | --- |
| `turn_cancel` | Durable cancel intent | Re-scan running turns with cancel intent after reconnect |
| `turn_activity` | A turn started or reached terminal | Re-read the subject-scoped activity snapshot |
| Host-command result channel | A browser posted a result for a paused host command | Poll the durable host-command row |

Each channel uses a dedicated reconnecting `LISTEN` connection. The notification is only a wake-up;
durable state remains the recovery source.

`GET /chat/activity` is a separate subject-scoped SSE feed. It emits a snapshot of running turns and
then live lifecycle transitions. It has no sequence, replay offset, or terminal event.
