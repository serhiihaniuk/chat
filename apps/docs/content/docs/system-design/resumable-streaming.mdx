---
title: Connection-Bound Streaming (ADR 0007)
description: Why generation is server-owned while live events remain bound to the instance that accepted the turn.
---

This page summarizes the accepted streaming decision. Read
[Assistant Turn Lifecycle](/docs/system-design/assistant-turn) for the ordered flow and
[Event Delivery](/docs/system-design/event-delivery) for the registry mechanics.

## Context

Generation must survive a browser disconnect. At the same time, persisting every model delta and
replaying it across instances adds storage, fan-out, retention, and correctness machinery that the
product does not need. The durable product result is the settled conversation, not the animation of
how text arrived.

## Decision

Use **server-owned generation with connection-bound live streaming**:

- `POST /chat/runs` runs setup, forks generation, and streams SSE on that same response.
- A per-instance in-memory registry buffers the live `sidechat.v1` events.
- `GET /chat/turns/:assistantTurnId/stream?after=<seq>` resumes only from the same registry.
- Postgres stores durable final state: conversation, messages, turn status, usage, context snapshots,
  and cancel intent.
- Owner leases plus a reaper recover turns whose process crashes before finalization.
- Cancel, turn activity, and host-command results use dedicated Postgres notification channels.
  Turn content does not.

## Recovery contract

Live resume is best effort outside the owning instance. The API makes that limit explicit:

| Condition | Response | Client action |
| --- | --- | --- |
| Owner still holds the registry entry | SSE replay from `after`, then live tail | Continue rendering |
| Turn is running on another instance | `409 stream_unavailable` | Poll durable status, then read history |
| Turn is terminal and no buffer exists | `404 replay_expired` | Read conversation history |

This split keeps the live path small while preserving the answer. A disconnect never cancels the
fiber. A process crash cannot preserve live deltas, but the reaper prevents the turn from remaining
`running` forever and history remains the convergence point.

## Alternatives rejected

- **Response-owned generation.** Tying model execution to the browser socket lets a disconnect
  abandon work or misclassify its outcome.
- **A durable cross-instance event log.** Persisting every delta would enable any-instance replay,
  but it adds a second durable representation of the answer plus notification, retention, and
  conflict rules.
- **A separate pub/sub system.** Redis does not solve the durable-result problem and adds another
  operational dependency.
- **A durable workflow engine.** It is unnecessary for the current long-running Node service and its
  bounded assistant-turn lifecycle.

## Consequences

- The service must run as a long-lived process because generation and live buffers are in memory.
- Initial streaming needs no sticky-routing feature: the POST response is already on the owner.
- Same-instance resume works from an exact sequence cursor.
- Cross-instance resume converges through status and history instead of pretending live replay is
  available.
- A reload may lose the typing animation while preserving the durable final answer.
- `partner-ai-core` keeps the `prepareStreamChatTurn` plus `runTurnGeneration` split so the service
  can finish setup before it forks generation.
