This is the decision record (ADR 0009, **accepted**) behind the streaming model. For the mechanics,
see [Assistant Turn Lifecycle](/docs/system-design/assistant-turn); for the narrative, see
[Life of a Turn](/docs/walkthrough/life-of-a-turn).

## Context

Chat streaming was response-owned: one `POST /chat/stream` request both ran the turn and carried its
Server-Sent Events. The in-flight turn lived only in that request and the browser's memory, so a tab
switch, reload, or dropped connection orphaned the turn — the browser lost the answer, and timestamps
drifted because persisted times came from the auth token's `issuedAt`. Users expect what Claude and
ChatGPT do: leave, come back, and the conversation is still there, still streaming.

## Decision

Make the assistant turn **server-owned and resumable**. Browser connections become disposable
subscribers that can reconnect from any instance and replay from a sequence offset.

- **Durable log as source of truth.** A `turn_events` table holds the ordered, append-only event log
  per turn. The browser-facing stream replays from this log, so events survive any disconnect.
- **Two-call HTTP flow.** `POST /chat/runs` runs pre-start synchronously and returns the turn identity
  as JSON; generation is then forked off the request. `GET /chat/turns/{assistantTurnId}/stream?after=<seq>`
  replays the log and tails live events. `assistantTurnId` is the canonical key; `requestId` is the
  idempotency and lost-reply resolver key only.
- **In-process generation.** Generation runs in server-owned Effect fibers (a turn runner with a
  `FiberMap` keyed by `assistantTurnId`), finalized through `Effect.onExit` so success, error, cancel,
  and shutdown all reach a durable terminal. A partial unique index enforces exactly one terminal per
  turn.
- **Postgres `LISTEN/NOTIFY` for liveness, no Redis.** The append writer notifies a channel in the
  same transaction; one dedicated `LISTEN` connection per instance fans signals out to local
  subscribers, with a low-frequency safety poll as the missed-signal backstop. Reads, not the signal,
  are the source of truth.
- **Crash recovery and retention.** Owner-lease fencing (`lease_epoch` CAS + heartbeat) plus a reaper
  terminalize dead- or slow-owner turns; a pruner bounds `turn_events` retention (a pruned turn falls
  back to conversation history on resume via `replay_expired`).
- **Cancel as durable intent.** Cancellation writes `cancel_requested_at` and notifies; the owning
  instance interrupts its fiber, and the reaper covers a dead owner.
- **Timestamps from the clock.** Persisted times come from `ports.clock.now()`, not the auth token.

## Alternatives rejected

- **Redis (pub/sub or streams)** for liveness or the event log — adds an operational dependency the
  durable Postgres log plus `LISTEN/NOTIFY` already make unnecessary.
- **Sticky routing** that pins a turn's stream to its generating instance — brittle under restarts and
  scaling; the durable log lets any instance serve any turn.
- **Keeping the response-owned stream** — the core defect; it cannot survive a disconnect.
- **A durable-execution worker** (a workflow engine) — out of scope for a long-running, horizontally
  scalable Node service; in-process fibers with lease recovery suffice.

## Consequences

- The service is a **long-running, horizontally scalable Node service**. Edge and serverless deployment
  are out of scope — generation outlives the request and relies on a held `LISTEN` connection and
  in-process fibers.
- `partner-ai-core` exposes the server-owned split `prepareStreamChatTurn` + `runTurnGeneration`; the
  response-owned `streamChatEffect` was removed.
- `db` owns the durable log, the dedicated `LISTEN/NOTIFY` connection, and the notification sources,
  behind the repository boundary.
- The subject-scoped activity stream (`GET /chat/activity`, channel `turn_activity`, `TurnActivityEvent`)
  reuses this same `NOTIFY` + fan-out + SSE pattern to push per-conversation "generating" status to the
  sidebar.
