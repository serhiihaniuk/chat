This page is the rendered reference for one assistant turn. It assumes you know the
[four-layer shape](/docs/system-design). For a narrative trace, see
[Life of a Turn](/docs/walkthrough/life-of-a-turn).

The browser sends a user message and an optional model preference. The service and Core supply the
trusted authority, profile, policy, instructions, tools, and context. The Runtime receives one
provider-neutral request after those decisions are complete.

## HTTP surface

| Call | Purpose | Result |
| --- | --- | --- |
| `POST /chat/runs` | Run pre-start, fork generation, register the turn before subscribing, and open the primary stream | SSE from `sidechat.started` through one terminal; pre-start failures are JSON |
| `GET /chat/turns/:assistantTurnId/stream?after=<seq>` | Resume a stream from the same instance | Registry replay for `sequence > after`, then live tail |
| `GET /chat/runs/:requestId` | Recover identity after a lost POST response | The existing `assistantTurnId` and status |
| `GET /chat/turns/:assistantTurnId` | Read a status snapshot | Turn identity and durable status |
| `POST /chat/turns/:assistantTurnId/cancel` | Record cancel intent and interrupt the owner | Idempotent cancel acknowledgement |

## Lifecycle

Stages 1–10 run synchronously before the response becomes SSE. Stages 11–14 run on the
server-owned generation fiber.

| # | Stage | Proves, records, or finalizes | Failure |
| ---: | --- | --- | --- |
| 1 | Validate the request | Method, authority middleware, JSON, and `ChatStreamRequest` | JSON `400` or auth error |
| 2 | Prove workspace authority | The subject may act in this workspace | Pre-start rejection |
| 3 | Record request received | Correlation and observation before runtime work | Pre-start rejection |
| 4 | Resolve the turn plan | Profile, model, reasoning, tools, executor, and instructions | Pre-start rejection |
| 5 | Run turn guards | Selected guards pass before private context or tools | Pre-start rejection |
| 6 | Ensure the conversation | The subject owns the loaded or created conversation | Pre-start rejection |
| 7 | Reject a concurrent turn | No assistant turn is already running for this conversation | `409 conversation_busy` |
| 8 | Append the user message | The visible message that starts the turn is durable | Pre-start rejection |
| 9 | Start the turn record | Status is `running`, idempotent on workspace plus `requestId` | Reject and mark failed |
| 10 | Prepare and record context | History, host context, tool context, and manifest snapshot | Reject and mark failed |
| 11 | Acquire the lease and emit `sidechat.started` | The owner is fenced and sequence 0 opens the stream | In-stream terminal |
| 12 | Execute the Runtime | The selected executor runs with an abortable provider request | In-stream terminal |
| 13 | Map and append live events | `RuntimeEvent` becomes `sidechat.v1` in the owner's registry | In-stream terminal |
| 14 | Finalize through `Effect.onExit` | Durable status, assistant message, usage, and optional title | Synthetic terminal when needed |

After stage 10, the service forks generation in a `FiberMap`, registers a fresh turn in its in-memory
registry before the POST response subscribes, and opens that response as SSE. A repeated `requestId`
resolves to the existing turn and never forks generation twice.

## Failure split

The split turns on whether the browser has seen `sidechat.started`:

| Phase | Started seen? | Behavior |
| --- | --- | --- |
| Pre-start | No | Reject setup as JSON |
| Post-start success | Yes | End with `sidechat.completed` |
| Post-start failure | Yes | End with `sidechat.error` or `sidechat.blocked` |

Finalization runs on success, provider failure, cancel, shutdown, and lease fencing. If the stream
already emitted a terminal, that terminal wins. If a fiber exits without one, finalization appends
one synthetic terminal to the registry before writing the durable failure state.

## Live transport

The turn-event registry is a per-instance, in-memory ordered buffer. The same object stores mapped
events and wakes local subscribers. It is not a database log and does not survive a process restart.

A subscriber registers before replay, reads buffered events with `sequence > after`, then tails a
bounded local queue. A dense sequence gate re-reads the buffer when a dropped queue offer creates a
gap. The stream closes at `completed`, `error`, or `blocked`. Once a terminal is recorded, the
registry ignores any later append. Terminal, unwatched entries are swept lazily.

## Resume and fallback

| Situation | Server response | Widget recovery |
| --- | --- | --- |
| Initial submit | POST response is SSE | Consume from `sidechat.started` |
| Same-instance reconnect | GET stream succeeds | Replay from the last sequence, then tail |
| Running turn on another instance | `409 stream_unavailable` | Poll status until terminal, then read history |
| Terminal turn whose buffer was swept | `404 replay_expired` | Read conversation history and clear the run |
| Terminal turn still buffered | GET replays through terminal | Close without tailing |

Postgres stores the final conversation, messages, turn record, status, usage, context snapshots,
cancel intent, and optional activity metadata. It does not store the in-flight turn-event stream.

## Cancel and crash recovery

Cancel is durable intent plus interruption. The route compare-and-set writes
`cancel_requested_at`, interrupts a local owner directly, and sends a `turn_cancel` notification so
a remote owner can interrupt its fiber. The interrupted provider request receives the abort signal;
finalization records `user_aborted` and protocol code `aborted`.

An owner lease and heartbeat fence each running turn to one instance. Every service instance also
runs a reaper. The reaper claims expired turns in disjoint batches, records an honest terminal state,
and advances the lease epoch so a stale owner cannot finalize later.

The remaining cross-instance notifications are limited to cancel, turn activity, and
host-command results. Turn content never uses Postgres `LISTEN/NOTIFY`.
